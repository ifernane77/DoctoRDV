Cette semaine correspond à une phase de cadrage : présentation du projet, définition du contexte, objectifs, backlog et dépôt GitHub.

### Travail réalisé

- Création de l’arborescence du projet.
- Préparation du README principal.
- Rédaction des objectifs du projet.
- Création du backlog initial.
- Initialisation du dépôt Git local.
- Création du dépôt GitHub.
- Préparation des premières preuves pour le portfolio.

### Difficultés rencontrées

J’ai dû structurer le projet de manière plus professionnelle afin qu’il soit exploitable pour l’épreuve E6 SLAM.

J’ai aussi clarifié la différence entre le dossier local, le dépôt Git local et le dépôt distant GitHub.

### Résultat obtenu

Le projet dispose maintenant d’une première base documentaire claire :

- présentation du projet ;
- contexte fictif ;
- objectifs ;
- backlog ;
- journal de bord ;
- dépôt GitHub.

### Preuves produites

## Semaine 2 — Acteurs, rôles et flux

### Objectif de la semaine

L’objectif de cette semaine était d’identifier les acteurs de l’application DoctoRDV et de comprendre les flux principaux autour de la prise de rendez-vous.

### Travail réalisé

- Identification des acteurs : visiteur, patient fictif, praticien et administrateur.
- Définition des rôles applicatifs.
- Analyse des flux de consultation, réservation, annulation et administration.
- Création d’un schéma simplifié des flux.
- Mise à jour de la documentation du projet.

### Difficultés rencontrées

La difficulté principale a été de garder une version simple de l’application. Certaines fonctionnalités pourraient être plus avancées, mais j’ai choisi de limiter le périmètre pour rester réaliste.

### Résultat obtenu

Le projet dispose maintenant d’une meilleure vision des utilisateurs et des interactions principales avec l’application.

### Preuves produites

- acteurs-et-roles.md
- architecture-et-flux.md
- capture du schéma Mermaid
- capture du commit Git

### Lien avec le BTS SIO

## Semaine 3 — Cas d’utilisation et architecture applicative

### Objectif de la semaine

L’objectif de cette semaine était de rédiger les cas d’utilisation principaux de DoctoRDV.

Ces cas d’utilisation permettent de préparer la conception fonctionnelle avant la création des maquettes, du MCD et du développement.

### Travail réalisé

- Rédaction des cas d’utilisation principaux.
- Création d’un diagramme de cas d’utilisation.
- Rédaction de l’architecture applicative prévue.
- Identification des données principales.
- Mise à jour de la documentation.

### Cas d’utilisation rédigés

- Rechercher un praticien.
- Voir les disponibilités.
- Réserver un rendez-vous.
- Annuler un rendez-vous.
- Consulter le planning praticien.
- Administrer les praticiens et les créneaux.

### Difficultés rencontrées

La difficulté principale a été de limiter le projet pour éviter de créer une application trop ambitieuse.

J’ai donc choisi une version simple avec des données fictives, sans paiement, sans SMS et sans données médicales sensibles.

### Résultat obtenu

Le projet dispose maintenant d’une base fonctionnelle claire pour préparer la suite : maquettes, MCD, base de données et développement.

### Preuves produites

- cas-utilisation.md
- diagramme-cas-utilisation.md
- architecture-application.md
- capture du diagramme Mermaid
- capture du commit Git

### Lien avec le BTS SIO

Cette semaine est directement liée à l’épreuve E6 SLAM, car elle prépare la conception d’une solution applicative.

## Semaine 4 — Préparation de l’environnement technique

### Objectif de la semaine

L’objectif de cette semaine était de préparer l’environnement technique du projet DoctoRDV avant de commencer le développement.

### Travail réalisé

- Création du fichier `installation.md`.
- Création du fichier `environnement-technique.md`.
- Création du fichier `conventions-nommage.md`.
- Préparation du dossier `database`.
- Création du fichier `database/README.md`.
- Création du fichier `database/schema-prevu.md`.
- Mise à jour de la documentation du projet.

### Difficultés rencontrées

J’ai compris qu’il ne fallait pas commencer directement par le code.

Le projet dispose maintenant d’une base technique préparée pour les prochaines étapes.

Le développement PHP, la création réelle de la base MySQL et les tests locaux seront réalisés plus tard.
